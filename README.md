# 《增删卜易》六爻占卜系统

本项目用确定性 Python 代码完成节气历、六爻排盘、规则事实与应期候选计算。模型先仅根据用户写下的“所占之事”和本地《增删卜易》原文判定用神，代码再把该选择定位到具体爻并计算元忌与应期。最终模型以理论原文确定规则、以完整卦例比较具体事项，再生成结构化断语。服务端会校验两个阶段的逐字引文、实例迁移、事实 ID 和应期候选，每个阶段失败时只允许模型纠正一次。

## 运行

需要 Python 3.12。

```bash
python -m venv .venv
source .venv/bin/activate
pip install -e '.[dev]'
cp .env.example .env
python scripts/build_knowledge_base.py
uvicorn app.main:app --host 127.0.0.1 --port 8000
```

网页位于 <http://127.0.0.1:8000/>。时间固定按 `Asia/Shanghai` 解释，默认以 23:00 为子初换日边界；请求中的六爻始终按初爻至上爻排列。

在 `.env` 中选择一个 `LLM_PROVIDER` 并只配置该供应商：

| Provider | 必需配置 |
|---|---|
| `openai` | `OPENAI_API_KEY`、`OPENAI_MODEL` |
| `azure_openai` | `AZURE_OPENAI_API_KEY`、`AZURE_OPENAI_ENDPOINT`、`AZURE_OPENAI_DEPLOYMENT`、`AZURE_OPENAI_API_VERSION` |
| `anthropic` | `ANTHROPIC_API_KEY`、`ANTHROPIC_MODEL` |

`LLM_TIMEOUT_SECONDS` 默认为 120 秒，可在 1–600 秒之间调整。SDK 隐式重试已关闭，超时会直接返回，不会在后台重复等待。系统不覆盖 Provider 的默认 reasoning effort。

## 系统设计：一次断卦如何完成

系统采用“**模型理解问题并选用神，代码定盘、定位与计算，理论定规则、实例作类比，模型再回答具体问题**”的信任边界。模型不能重新排盘、添加规则或自行计算应期；断卦阶段也不能推翻前置模型已经作出的用神判定。

```text
浏览器输入
   ↓
请求校验 → 用神判定模型（只看所占之事与用神原文）
   ↓
确定性历法 → 六爻排盘 → 用神爻定位 → 规则事实 → 应期候选
   ↓
SQLite 分流检索理论原文与完整卦例
   ↓
受约束断卦模型完成“本问映射 → 卦例比照 → 吉凶综合”
   ↓
事实、用神、实例迁移、应期和逐字引文校验
   ├─ 通过 → 返回结果
   └─ 失败 → 带具体错误纠正一次 → 再失败则拒绝输出
```

### 1. 输入与模型判类

页面只要求所占之事、公历时间和恰好六个 `6/7/8/9` 爻值。占类、用神以及“自占/代占”不再由额外控件提供；用户应直接在所占之事中写清楚所问的人、事及关系。后端不信任浏览器计算结果，并拒绝旧的 `category`、`subject` 等额外请求字段。

完整断卦的第一次结构化模型调用只收到：

- 用户原样输入的所占之事；
- 父母、官鬼、兄弟、妻财、子孙、世爻及天时用神的指定原文；
- 固定的占类、用神方式和六亲枚举；
- `UsefulGodDecision` JSON Schema。

模型必须输出占类、问题中的实际对象、`world`（世爻）或 `relative`（六亲）方式、所选六亲、理由和至少一条逐字引文。服务端检查出处和引文后才进入排盘规则阶段；模型不能假定页面中另有占问者身份、性别或关系信息。

### 2. 历法和排盘

历法模块固定按 `Asia/Shanghai` 解释时间：

- 使用固定版本 `lunar-python==1.4.8` 的精确节气时刻，以“节”而非农历初一切换月建；
- 立春切换干支年；
- 默认 23:00 子初换日，计算日柱、时柱、旬空和六神；
- 临近交节或换日边界时，在结果中给出审计说明。

排盘模块随后完全由代码完成：

1. `6/8` 为阴、`7/9` 为阳，`6/9` 为动爻；
2. 识别主卦，翻转动爻后识别变卦；
3. 查定六十四卦名称、八宫归属、宫序、宫五行、世应；
4. 逐爻安装纳甲干支、五行、六亲和六神；
5. 变爻的六亲仍以主卦宫五行为基准；
6. 用神不上卦时，从本宫首卦计算伏神；
7. 标记六冲、六合、游魂和归魂。

每项结果都形成稳定事实 ID，例如 `fact-month-break-l3`，同时保留计算参数和原文章节来源。

### 3. 用神如何选择与定位

用神角色由前置模型根据用户输入判定，不再由下拉框、单选框或关键词表决定。模型可以选择：

| 模式 | 含义 |
|---|---|
| `world` | 问题所指是问占者本人整体事项，代码采用本卦世爻 |
| `relative` | 问题指向原文规定的人物或事物，模型在父母、兄弟、官鬼、妻财、子孙中选择一个 |

模型只决定“取世爻还是哪个六亲”，不直接修改排盘或任意指定爻位。代码收到判定后寻找世爻或相应六亲；若同一六亲在卦中多现，候选仍按以下固定分值排序：

| 条件 | 分值 |
|---|---:|
| 动爻 | +30 |
| 月建 / 旺 / 相 / 余气 | +25 / +20 / +15 / +5 |
| 旬空 / 月破 | -5 / -5 |
| 临世 / 临应 | +3 / +2 |

最高分并列时，系统保留 `multiple` 状态，不暗中采用第一爻，也不生成应期。卦中不现时依次检查伏神、月建和日辰；仍无法定位则返回 HTTP 409。确定用神五行后，代码再计算：

- 元神：生用神者；
- 忌神：克用神者；
- 仇神：克元神者。

### 4. 规则引擎计算什么

规则引擎只输出可审计事实，不直接写自由文本吉凶结论。目前包括：

- 四时旺相休囚、月建、日辰作用；
- 旬空、月破、日冲、暗动、动散；
- 地支冲合刑、月合、日合；
- 动变五行关系、回头生克、回头冲合；
- 长生、帝旺、墓、绝；
- 进神、退神；
- 飞神与伏神关系；
- 三合局、六冲、六合；
- 反吟、伏吟、独发、两现；
- 前置模型所选用神、元神、忌神、仇神及其动爻作用。

原书明确称六害“全无应验”，因此系统不把六害当作预测证据。复杂随鬼入墓、复杂飞伏权衡和多卦合参等未实现部分会明确列在响应的 `limitations` 中，不会转交模型自由发挥。

### 5. 应期如何产生

应期同样先由代码生成候选。只有确定到具体用神爻时，系统才会根据其事实产生候选，例如：

- 静爻逢值、逢冲；
- 动爻逢值、逢合；
- 旬空填实、冲空；
- 月破填实、逢合；
- 入墓冲开、六合待冲；
- 太旺待墓或冲、衰绝待生旺；
- 忌神受制；
- 进神、退神的值、合、冲。

候选只包含地支、远近时间单位、适用限制、事实 ID 和原文 ID。系统不把它自动换算成绝对日期；LLM 也只能从这些候选中选择，不能新增日期。

### 6. 原文知识库和检索

141 章 Markdown 在构建时被解析为稳定段落和卦例，写入 SQLite + FTS5。段落保存 `source_id`、原文、标签、源文件路径、字符偏移和 SHA-256；`GET /api/v1/sources/{source_id}` 会回到当前 Markdown 校验原文。

理论段落和卦例分开检索，卦例不会再拆散后混入普通理论。一次断卦按以下顺序处理：

1. 前置模型用神判定、本次排盘、规则和应期实际引用的段落，强制加入；
2. 从非实例规则中取凡例、用神和通用方法论，最多 20 段；
3. 从非实例规则中取对应占类章节，最多 18 段；
4. 从非实例规则中取与本次事实标签对应的段落，最多 18 段；
5. FTS5 只补充非实例规则，最多 8 段；
6. 只保留同时具有原占问、原卦盘、非编辑性原断语的完整卦例；
7. 按占类专属性、本卦事实标签、主卦与变卦、用神六亲以及用户问题文字评分，只提供最相关的 1 个完整实例。

同占类的专门章节优先于仅有若干相同术语的异类案例，避免求财问题被寻地、疾病案例挤占。每个候选以 `example_id` 将原占问、卦盘、断语和匹配理由成组交给模型，而不是三段无关系的文本。结果按稳定顺序去重；`[乾按]`、`[提要]`等编辑性段落不会作为模型判断依据。任何强制规则出处缺失或仅指向编辑按语时，服务直接失败，不带病调用模型。

### 7. 两次 LLM 调用分别做什么

第一次调用只做占类与用神判定，不接收排盘，也不生成吉凶断语。第二次调用只收到：

- 原问题和前置模型判定的占类；
- 完整结构化排盘；
- 前置模型所选、代码定位后的用神；
- 代码事实及其稳定 ID；
- 代码生成的应期候选；
- 与实例分开的规则、理论原文；
- 1 个带原占问、卦盘、原断语和匹配理由的最相关卦例；
- 强制 JSON Schema。

第二次调用先在 `question_application` 中把旺衰、空破、生克、动变等术语落实为用户所问对象的具体得失，再在 `case_analysis` 中比较最相关的一个完整卦例，分别说明相似点、关键差异和哪些原断可以迁移，最后综合成总体趋势、用神分析、应期及风险。实例不是按卦名机械套用：相似点、差异和迁移结论都必须同时引用该例原文与本卦事实，其中迁移结论必须引用原例断语。

证据明显偏向一方时，模型必须在“吉、凶、平”中给出方向。没有逐字对应现代事项的古文、没有完全相同的卦例，不能单独成为“不确定”的理由；只有本卦有利与不利事实直接冲突且原文、实例都无法分出主次时才允许“不确定”。页面会同时展示具体问题映射、模型实际采用的卦例、未采用候选、匹配理由、相似点、差异、迁移判断及完整原例。

### 8. 返回前如何防止模型胡说

服务端逐项检查：

- 第一次调用的用神出处是否在允许集合中、引文是否逐字一致；
- 每条判断是否带有证据；
- `fact_id` 是否属于本次排盘；
- `source_id` 是否属于本次检索；
- 引文是否为对应段落的逐字子串；
- 第二次调用是否擅自改变前置模型所选用神或代码定位的爻位；
- 是否在有候选卦例时完全跳过实例比照；
- `example_id` 是否属于本轮候选，相似点、差异和迁移结论是否同时连接原例与本卦事实；
- 迁移结论是否引用对应卦例的原断语，而非只引用题目或卦盘；
- 对用户具体问题的综合判断是否至少引用一个本卦事实；
- “不确定”是否确有同时列明、无法分出主次的有利与不利证据；
- 对某爻空、破、动、静、旺、衰、生、克的陈述是否有匹配的爻位事实；
- 应期是否来自已选代码候选，是否擅自增加地支或绝对日期；
- 是否混入其他术数流派或禁止术语。

每个模型阶段首次失败时都会收到具体错误并完整纠正一次；同一阶段第二次仍不合格则返回 `llm_validation_failed`，不会继续使用未经验证的用神判定或断语。

### 9. 为什么完整断卦比排盘慢

确定性部分很快。当前求财示例的一次本机测量：

| 阶段 | 耗时 |
|---|---:|
| 历法、排盘、规则 | 约 10 ms |
| 原文检索 | 约 37 ms |
| 上下文与提示词构建 | 约 6 ms |
| 本地合计 | 约 53 ms |

当前断卦阶段不再把全部路由结果直接塞给模型：强制规则出处之外，只保留少量通用规则、占类规则、事实规则和一个最相关完整卦例；结构化 Schema 由 Provider 协议传递，不在用户消息中重复一份。标准求财样例的断卦消息已从约 3.1 万字符压缩到 1.5 万字符以内。完整流程仍包含用神判定和最终断卦两次模型调用；任一阶段未通过校验时会增加该阶段的一次纠正调用，且每次调用分别受 `LLM_TIMEOUT_SECONDS` 限制。`POST /api/v1/chart` 不调用模型，通常可立即返回，但不会给出用神、元忌或应期。

## API

| 方法 | 路径 | 行为 |
|---|---|---|
| `GET` | `/` | 原生 HTML/JavaScript 页面 |
| `GET` | `/healthz` | 健康检查 |
| `POST` | `/api/v1/chart` | 仅执行确定性历法、排盘和通用规则计算，不需要知识库或 LLM 凭据，不判定用神 |
| `POST` | `/api/v1/divinations` | 模型判定用神、代码定位与计算、检索原文、模型断卦并校验后返回 |
| `GET` | `/api/v1/sources/{source_id}` | 返回稳定引用对应的本地原文 |

用神两现且评分相同时保留歧义，不擅自选择第一爻，也不生成应期候选；若模型所选六亲在当前排盘、伏神、月建和日辰中均无法定位，接口返回 HTTP 409。知识库或当前 Provider 配置不可用时会明确返回 503，不会切换供应商。

## 知识库

原始语料位于 `zengshan_buyi/`，构建产物写入 `data/generated/`，构建过程不会修改原始 Markdown。当前构建结果为 141/141 章、2,251 个段落、321 个卦例和 191 个编辑性段落；27 条非致命解析警告记录在 `data/generated/knowledge_build_report.json`。

```bash
python scripts/build_knowledge_base.py
python scripts/extract_examples.py
```

## 明确限制

API 响应会列出未实现或不作确定判断的规则。首版不把六害作为吉凶依据（原书明确“全无应验”），也不自动处理随鬼入墓的语义适用条件、复杂飞伏权衡、多卦合参或门类应期的绝对日期换算；这些内容不会交给 LLM 自行补算。
