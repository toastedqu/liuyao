# 《增删卜易》六爻占卜系统

本项目用确定性 Python 代码完成节气历、六爻排盘、用神、规则事实与应期候选计算。LLM 只能根据本次排盘事实和本地《增删卜易》原文生成结构化断语；服务端会校验事实 ID、逐字引文和应期候选，失败时只允许模型纠正一次。

## 运行

需要 Python 3.12。

```bash
python -m venv .venv
source .venv/bin/activate
pip install -e '.[dev]'
cp .env.example .env
python scripts/build_knowledge_base.py
uvicorn app.main:app --host 127.0.0.1 --port 8000
```

网页位于 <http://127.0.0.1:8000/>。时间固定按 `Asia/Shanghai` 解释，默认以 23:00 为子初换日边界；请求中的六爻始终按初爻至上爻排列。

在 `.env` 中选择一个 `LLM_PROVIDER` 并只配置该供应商：

| Provider | 必需配置 |
|---|---|
| `openai` | `OPENAI_API_KEY`、`OPENAI_MODEL` |
| `azure_openai` | `AZURE_OPENAI_API_KEY`、`AZURE_OPENAI_ENDPOINT`、`AZURE_OPENAI_DEPLOYMENT`、`AZURE_OPENAI_API_VERSION` |
| `anthropic` | `ANTHROPIC_API_KEY`、`ANTHROPIC_MODEL` |

`LLM_TIMEOUT_SECONDS` 默认为 120 秒，可在 1–600 秒之间调整。SDK 隐式重试已关闭，超时会直接返回，不会在后台重复等待。

## 系统设计：一次断卦如何完成

系统采用“**代码定盘与计算，LLM 只负责受约束的综合断语**”的信任边界。模型不能重新排盘、改选用神、添加规则或自行计算应期。

```text
浏览器输入
   ↓
请求校验与占类确认
   ↓
确定性历法 → 六爻排盘 → 用神/规则事实 → 应期候选
   ↓
SQLite 分层检索《增删卜易》原文
   ↓
受约束 LLM 生成结构化断语
   ↓
事实、用神、应期和逐字引文校验
   ├─ 通过 → 返回结果
   └─ 失败 → 带具体错误纠正一次 → 再失败则拒绝输出
```

### 1. 输入和占类

后端重新校验问题、公历时间、自占/代占关系，以及恰好六个 `6/7/8/9` 爻值，不信任浏览器计算结果。

- 用户明确选择占类时直接采用。
- 未选择时，由代码按问题关键词评分；唯一最高分可自动确认。
- 并列、没有可靠匹配或问题对象不足以确定用神时返回 HTTP 409，请用户明确选择，而不是让 LLM 猜测。
- 代占关系也由代码映射为父母、兄弟、妻财、官鬼或子孙。

### 2. 历法和排盘

历法模块固定按 `Asia/Shanghai` 解释时间：

- 使用固定版本 `lunar-python==1.4.8` 的精确节气时刻，以“节”而非农历初一切换月建；
- 立春切换干支年；
- 默认 23:00 子初换日，计算日柱、时柱、旬空和六神；
- 临近交节或换日边界时，在结果中给出审计说明。

排盘模块随后完全由代码完成：

1. `6/8` 为阴、`7/9` 为阳，`6/9` 为动爻；
2. 识别主卦，翻转动爻后识别变卦；
3. 查定六十四卦名称、八宫归属、宫序、宫五行、世应；
4. 逐爻安装纳甲干支、五行、六亲和六神；
5. 变爻的六亲仍以主卦宫五行为基准；
6. 用神不上卦时，从本宫首卦计算伏神；
7. 标记六冲、六合、游魂和归魂。

每项结果都形成稳定事实 ID，例如 `fact-month-break-l3`，同时保留计算参数和原文章节来源。

### 3. 用神如何选择

用神不是由模型选择。代码首先按占类、关系和问题中的明确对象确定六亲：

| 情形 | 用神 |
|---|---|
| 求财 | 妻财 |
| 功名 | 官鬼 |
| 胎产 | 子孙 |
| 家宅、茔葬、学业 | 父母 |
| 雨、雪、阴天 | 父母 |
| 晴、放晴 | 子孙 |
| 雷霆 | 官鬼 |
| 自占身命、出行、疾病、诉讼 | 世爻 |
| 明确父母、配偶、子女等对象 | 对应六亲 |

若同一六亲在卦中多现，候选按以下固定分值排序：

| 条件 | 分值 |
|---|---:|
| 动爻 | +30 |
| 月建 / 旺 / 相 / 余气 | +25 / +20 / +15 / +5 |
| 旬空 / 月破 | -5 / -5 |
| 临世 / 临应 | +3 / +2 |

最高分并列时，系统保留 `multiple` 状态，不暗中采用第一爻，也不生成应期。卦中不现时依次检查伏神、月建和日辰；仍不能确定则返回 HTTP 409。

确定用神五行后，代码再计算：

- 元神：生用神者；
- 忌神：克用神者；
- 仇神：克元神者。

### 4. 规则引擎计算什么

规则引擎只输出可审计事实，不直接写自由文本吉凶结论。目前包括：

- 四时旺相休囚、月建、日辰作用；
- 旬空、月破、日冲、暗动、动散；
- 地支冲合刑、月合、日合；
- 动变五行关系、回头生克、回头冲合；
- 长生、帝旺、墓、绝；
- 进神、退神；
- 飞神与伏神关系；
- 三合局、六冲、六合；
- 反吟、伏吟、独发、两现；
- 用神、元神、忌神、仇神及其动爻作用。

原书明确称六害“全无应验”，因此系统不把六害当作预测证据。复杂随鬼入墓、复杂飞伏权衡和多卦合参等未实现部分会明确列在响应的 `limitations` 中，不会转交模型自由发挥。

### 5. 应期如何产生

应期同样先由代码生成候选。只有确定到具体用神爻时，系统才会根据其事实产生候选，例如：

- 静爻逢值、逢冲；
- 动爻逢值、逢合；
- 旬空填实、冲空；
- 月破填实、逢合；
- 入墓冲开、六合待冲；
- 太旺待墓或冲、衰绝待生旺；
- 忌神受制；
- 进神、退神的值、合、冲。

候选只包含地支、远近时间单位、适用限制、事实 ID 和原文 ID。系统不把它自动换算成绝对日期；LLM 也只能从这些候选中选择，不能新增日期。

### 6. 原文知识库和检索

141 章 Markdown 在构建时被解析为稳定段落和卦例，写入 SQLite + FTS5。段落保存 `source_id`、原文、标签、源文件路径、字符偏移和 SHA-256；`GET /api/v1/sources/{source_id}` 会回到当前 Markdown 校验原文。

一次断卦按以下顺序检索：

1. 本次排盘、规则、用神和应期实际引用的段落，强制加入；
2. 凡例、用神和通用方法论，最多 20 段；
3. 对应占类章节，最多 18 段；
4. 与本次事实标签对应的规则，最多 18 段；
5. FTS5 关键词补充，最多 8 段；
6. 按占类、事实标签和卦名评分的卦例，最多 4 例。

结果按稳定顺序去重；`[乾按]`、`[提要]`等编辑性段落不会作为模型判断依据。任何强制规则出处缺失或仅指向编辑按语时，服务直接失败，不带病调用模型。

### 7. LLM 到底做什么

模型只收到：

- 原问题、已确认占类和占问关系；
- 完整结构化排盘；
- 代码选定的用神；
- 代码事实及其稳定 ID；
- 代码生成的应期候选；
- 本次检索到的《增删卜易》原文；
- 强制 JSON Schema。

它的职责是把这些材料综合成总体趋势、用神分析、月建日辰、动爻与元忌、特殊格局、应期选择及风险限制。每条判断必须引用事实 ID 和/或原文；引用必须逐字摘录，不能转述成“原文”。

### 8. 返回前如何防止模型胡说

服务端逐项检查：

- 每条判断是否带有证据；
- `fact_id` 是否属于本次排盘；
- `source_id` 是否属于本次检索；
- 引文是否为对应段落的逐字子串；
- 用神六亲和爻位是否与代码结果一致；
- 对某爻空、破、动、静、旺、衰、生、克的陈述是否有匹配的爻位事实；
- 应期是否来自已选代码候选，是否擅自增加地支或绝对日期；
- 是否混入其他术数流派或禁止术语。

首次失败时，模型会收到具体错误并完整纠正一次。第二次仍不合格则返回 `llm_validation_failed`，不会把未经验证的断语展示给用户。

### 9. 为什么完整断卦比排盘慢

确定性部分很快。当前求财示例的一次本机测量：

| 阶段 | 耗时 |
|---|---:|
| 历法、排盘、规则 | 约 10 ms |
| 原文检索 | 约 37 ms |
| 上下文与提示词构建 | 约 6 ms |
| 本地合计 | 约 53 ms |

该示例最终包含 52 条事实、92 段原文和约 3.1 万字符提示词；真实 Azure OpenAI 结构化生成约耗时 98 秒。若首轮结果未通过校验，还会进行一次完整纠正调用，因此总时间可能超过单次 `LLM_TIMEOUT_SECONDS`。`POST /api/v1/chart` 不调用模型，通常可立即返回。

## API

| 方法 | 路径 | 行为 |
|---|---|---|
| `GET` | `/` | 原生 HTML/JavaScript 页面 |
| `GET` | `/healthz` | 健康检查 |
| `GET` | `/api/v1/categories` | 支持的占类、关系和固定时区 |
| `POST` | `/api/v1/chart` | 仅执行确定性历法、排盘和规则计算，不需要知识库或 LLM 凭据 |
| `POST` | `/api/v1/divinations` | 检索原文、调用当前 Provider、校验并返回完整断卦 |
| `GET` | `/api/v1/sources/{source_id}` | 返回稳定引用对应的本地原文 |

占类存在歧义，或问题不足以确定用神时，完整断卦接口返回 HTTP 409，并且不会调用 LLM。用神两现且评分相同时保留歧义，不擅自选择第一爻，也不生成应期候选。知识库或当前 Provider 配置不可用时会明确返回 503，不会切换供应商。

## 知识库

原始语料位于 `zengshan_buyi/`，构建产物写入 `data/generated/`，构建过程不会修改原始 Markdown。当前构建结果为 141/141 章、2,251 个段落、321 个卦例和 191 个编辑性段落；27 条非致命解析警告记录在 `data/generated/knowledge_build_report.json`。

```bash
python scripts/build_knowledge_base.py
python scripts/extract_examples.py
```

## 明确限制

API 响应会列出未实现或不作确定判断的规则。首版不把六害作为吉凶依据（原书明确“全无应验”），也不自动处理随鬼入墓的语义适用条件、复杂飞伏权衡、多卦合参或门类应期的绝对日期换算；这些内容不会交给 LLM 自行补算。
