# 《增删卜易》六爻占卜系统

本项目用确定性 Python 代码完成节气历、六爻排盘、用神爻定位、规则事实与应期候选计算。用户明确选择世爻、应爻或一种六亲作为用神，代码再依据《增删卜易》处理两现、不上卦和伏神并计算元忌与应期；模型只提取占类及自占/代占视角，无权改选用神。最终模型只以本卦用神及有效元忌的裁决证据决定方向，完整卦例仅在主判断完成后作为方法参考。服务端会校验裁决证据方向、卦例隔离、事实 ID、应期候选及逐字引文，最终断卦首次失败时只允许模型纠正一次。

## 运行

需要 Python 3.12。

```bash
python -m venv .venv
source .venv/bin/activate
pip install -e '.[dev]'
cp .env.example .env
python scripts/build_knowledge_base.py
uvicorn app.main:app --host 127.0.0.1 --port 8000
```

网页位于 <http://127.0.0.1:8000/>。时间固定按 `Asia/Shanghai` 解释，默认以 23:00 为子初换日边界；请求中的六爻始终按初爻至上爻排列。

在 `.env` 中选择一个 `LLM_PROVIDER` 并只配置该供应商：

| Provider | 必需配置 |
|---|---|
| `openai` | `OPENAI_API_KEY`、`OPENAI_MODEL` |
| `azure_openai` | `AZURE_OPENAI_API_KEY`、`AZURE_OPENAI_ENDPOINT`、`AZURE_OPENAI_DEPLOYMENT`、`AZURE_OPENAI_API_VERSION` |
| `anthropic` | `ANTHROPIC_API_KEY`、`ANTHROPIC_MODEL` |

`LLM_TIMEOUT_SECONDS` 默认为 120 秒，可在 1–600 秒之间调整。SDK 隐式重试已关闭，超时会直接返回，不会在后台重复等待。系统不覆盖 Provider 的默认 reasoning effort。

## 系统设计：一次断卦如何完成

系统采用“**用户选用神，模型只提取占类与自占/代占视角，代码定盘、定位并筛出裁决证据，理论解释权衡，卦例只作参考，模型回答具体问题**”的信任边界。模型不能改选用神、重新排盘、添加规则或自行计算应期。

```text
浏览器输入问题、用神、时间和六爻
   ↓
请求校验 → 语义模型（只提取占类与自占/代占）
   ↓
确定性历法 → 六爻排盘 → 用神爻定位 → 规则事实 → 吉凶裁决证据 → 应期候选
   ↓
SQLite 分流检索理论原文与完整卦例
   ↓
受约束断卦模型完成“本卦主判断 → 本问映射 → 卦例参考”
   ↓
事实、用户所选用神、裁决方向、卦例隔离、应期和逐字引文校验
   ├─ 通过 → 返回结果
   └─ 失败 → 带具体错误纠正一次 → 再失败则拒绝输出
```

### 1. 输入与模型判类

页面要求所占之事、用神、公历时间和恰好六个 `6/7/8/9` 爻值。用神可选世爻、应爻、父母、兄弟、官鬼、妻财或子孙；用户不选择具体爻位。后端不信任浏览器计算结果，并拒绝 `category`、`subject`、`useful_god_line` 等额外请求字段。

完整断卦的第一次结构化模型调用只收到用户原样输入的所占之事、固定占类枚举和自占/代占枚举，必须通过 `QuestionCategory` 选择一个占类和问占视角。它不会收到“选择用神”的任务，也不能覆盖用户提交的用神。恋爱、择偶、夫妻关系归入婚姻；用户则根据实际问占对象自行选择妻财、官鬼等用神。自占/代占只用于《随鬼入墓章》等明确区分“自占看世、代占看用”的规则，不改变用户所选用神。

### 2. 历法和排盘

历法模块固定按 `Asia/Shanghai` 解释时间：

- 使用固定版本 `lunar-python==1.4.8` 的精确节气时刻，以“节”而非农历初一切换月建；
- 立春切换干支年；
- 默认 23:00 子初换日，计算日柱、时柱、旬空和六神；
- 临近交节或换日边界时，在结果中给出审计说明。

排盘模块随后完全由代码完成：

1. `6/8` 为阴、`7/9` 为阳，`6/9` 为动爻；
2. 识别主卦，翻转动爻后识别变卦；
3. 查定六十四卦名称、八宫归属、宫序、宫五行、世应；
4. 逐爻安装纳甲干支、五行、六亲和六神；
5. 变爻的六亲仍以主卦宫五行为基准；
6. 用神不上卦时，从本宫首卦计算伏神；
7. 标记六冲、六合、游魂和归魂。

每项结果都形成稳定事实 ID，例如 `fact-month-break-l3`，同时保留计算参数和原文章节来源。

### 3. 用神如何选择与定位

用神角色由用户明确选择，模型不能更改。用户可以选择：

| 模式 | 含义 |
|---|---|
| `世爻` | 以问占者本人或主事者为用 |
| `应爻` | 以对方或原文明确规定的应爻为用 |
| 五类六亲 | 父母、兄弟、官鬼、妻财、子孙 |

代码负责具体爻位。六亲只现一处时直接采用；两现或多现时，不使用人为加权分数，而按《两现章》所列古法依次比较月令旺衰、动静、不破、不空和不受伤。若只有一个候选优先，系统标记其为默认主用爻；但每个候选都会分别完成旺衰、生克、空破、效力和应期裁决，再合并方向，因为野鹤给出了舍旺取空、舍不破取月破的反例。候选方向一致时可据共同方向判断；方向不同或证据不足且没有前卦可供合参时输出“需再占”，不会要求用户选择爻位。

用神不上卦时，代码依《飞伏神章》先检查月建、日辰是否就是该六亲；日月不是用神时，再检查变爻，变爻仍无才取本宫首卦伏神并计算伏神自身的旺衰、空破和飞伏效力。用神只现于变爻时，以变爻自身的地支、五行、旺衰、日辰关系、空破、生克与应期独立裁决，不拿本爻状态冒充变爻状态。只有日月、本卦、变爻和本宫伏神都不能定位时才返回 HTTP 409，并明确建议另占，而不是让用户猜一爻。确定用神五行后，代码再计算：

- 元神：生用神者；
- 忌神：克用神者；
- 仇神：克元神者。

### 4. 规则引擎计算什么

规则引擎只输出可审计事实，不直接写自由文本吉凶结论。目前包括：

- 四时旺相休囚、月建、日辰作用；
- 旬空、月破、日冲、暗动、动散；
- 地支冲合刑、月合、日合；
- 动变五行关系、回头生克、回头冲合；
- 长生、帝旺、墓、绝；
- 进神、退神；
- 飞神与伏神关系；
- 三合局、六冲、六合；
- 反吟、伏吟、独发、两现；
- 六神只能附和既有五行方向，不能独立造吉造凶；
- 合中带克、讼狱逢合及六冲变六合；
- 太乙贵人、禄神、驿马、天喜只在用神旺而有扶时作为辅证；
- 太岁只记录当年背景，不凭术语定义增加作用力；
- 用户所选并由代码定位的用神、元神、忌神、仇神及其动爻作用。

原书明确称六害“全无应验”，并删弃四煞之外的大量星煞，因此系统不把它们当作预测证据。随鬼入墓按自占看世、代占看用执行；卦化墓绝、太岁独立作用力等原文没有可执行判据的部分会明确列在响应的 `limitations` 中，不会转交模型自由发挥。

规则事实之后还有一层保守的“吉凶裁决证据”：

- 只纳入已定位用神的月日旺衰、空破、动变，以及确实发动或暗动并直接生克用神的元神、忌神；
- 将证据标成“有利 / 不利 / 条件性 / 仅背景”和“主证 / 辅证”；
- 元神、忌神仅仅出现在卦中不计吉凶，主卦名、六冲、六合、游魂、归魂也不单独定向；
- 动而月破、旺动逢空等先列为时间条件，不自动贴“凶”标签；
- 元忌同动时先检查贪生忘克；用神休囚、月破并受日克时先检查“无根”，不能用一个元神机械翻吉；
- 不做任意数值相加。只有单向主证且没有未决条件时才形成“仅有利主证”或“仅不利主证”；正反并见或条件未决时由模型依据原文说明主次，代码不伪造精确分数。

### 5. 应期如何产生

应期同样先由代码生成候选。只有确定到具体用神爻时，系统才会根据其事实产生候选，例如：

- 静爻逢值、逢冲；
- 动爻逢值、逢合；
- 旬空填实、冲空；
- 月破填实、逢合；
- 入墓冲开、六合待冲；
- 太旺待墓或冲、衰绝待生旺；
- 忌神受制；
- 进神、退神的值、合、冲。

候选只包含地支、远近时间单位、适用限制、事实 ID 和原文 ID。系统不把它自动换算成绝对日期；LLM 也只能从这些候选中选择，不能新增日期。

### 6. 原文知识库和检索

141 章 Markdown 在构建时被解析为稳定段落和卦例，写入 SQLite + FTS5。只有同时包含六爻线图和卦宫标记、且不是“假令占得”等教学示意的 fenced block 才会被识别为占验卦例；第 035 章飞伏神示意图和第 010 章元忌规则清单均保留为理论。卦例断语会延伸到下一卦或章节边界，再占桥接段同时归入前卦断语与后卦问题。段落保存 `source_id`、原文、标签、源文件路径、字符偏移和 SHA-256；`GET /api/v1/sources/{source_id}` 会回到当前 Markdown 校验原文。

理论段落和卦例分开检索，卦例不会再拆散后混入普通理论。一次断卦按以下顺序处理：

1. 用户所选用神的映射出处、本次排盘、规则和应期实际引用的段落，强制加入；
2. 所有激活规则均按规则注册表中的 `source_id` 精确回源，不再依赖固定章节或事实标签搜索补规则；
3. 从非实例规则中取对应占类章节，最多 12 段；
4. 以用户完整问题执行 FTS5，只补充非实例规则，最多 6 段；
5. 只保留同时具有原占问、原卦盘、非编辑性原断语的完整卦例；
6. 按占类专属性、本卦事实标签、主卦与变卦、用神六亲以及用户问题文字做检索排序，提供最多 3 个完整实例；该分数只决定展示哪些参考例，不代表吉凶相似度。

同占类的专门章节优先于仅有若干相同术语的异类案例，避免求财问题被寻地、疾病案例挤占。每个候选以 `example_id` 将原占问、卦盘和断语成组交给模型，但不向模型提供检索分或匹配理由，避免产生“越像就越应同吉凶”的锚定。结果按稳定顺序去重；`[乾按]`、`[提要]`等编辑性段落不会作为模型判断依据。任何强制规则出处缺失或仅指向编辑按语时，服务直接失败，不带病调用模型。

### 7. 两次 LLM 调用分别做什么

第一次调用只提取一个占类和自占/代占视角，不接收排盘、不选择用神，也不生成吉凶断语。代码随后采用用户提交的用神并依据《增删卜易》定位具体爻位。第二次调用只收到：

- 原问题、模型判定的占类和自占/代占视角；
- 完整结构化排盘；
- 用户选择并由代码定位后的用神；
- 代码事实及其稳定 ID；
- 仅与用神及有效元忌直接相关的裁决证据、证据方向、层级和保守质量控制结论；
- 代码生成的应期候选；
- 与实例分开的规则、理论原文；
- 最多 3 个带原占问、卦盘、完整原断语但不带匹配分的参考卦例；
- 强制 JSON Schema。

第二次调用必须先完全不依赖卦例结果，只按本卦裁决证据与理论原文形成 `overall` 和 `question_application`。随后才在 `case_analysis` 中以固定 `role=reference_only` 比较一个完整卦例，说明相似点、差异和方法适用边界；该部分不得支持、推翻或回改主判断。

总体结论采用机器强制集合：代码只发现有利主证时只能输出“吉”，只发现不利主证时只能输出“凶”，正反并见时只能输出“吉中有阻”或“凶中有救”，裁决层保留时只能输出“需再占”。模型不能输出“平”或“不确定”绕过裁决，也不能因缺少完全相同的卦例改变允许集合。页面会同时展示裁决证据、具体问题映射、参考卦例、检索原因、参考边界及完整原例，并明确检索分不是吉凶分。

### 8. 返回前如何防止模型胡说

服务端逐项检查：

- 每条判断是否带有证据；
- `fact_id` 是否属于本次排盘；
- `source_id` 是否属于本次检索；
- 引文是否为对应段落的逐字子串；
- 第二次调用是否擅自改变用户所选用神或代码定位的爻位；
- 是否在有候选卦例时完全跳过参考比照，`role` 是否固定为 `reference_only`；
- `example_id` 是否属于本轮候选，参考比照是否同时连接原例与本卦事实；
- 卦例出处是否越界进入总体结论、问题综合、用神、月日或动爻分析；
- 对用户具体问题的综合判断是否引用本卦裁决证据，而非普通卦名或元忌“出现”事实；
- 有利、不利列表是否分别只引用相同方向的裁决证据；
- `overall.outlook` 是否属于裁决层给出的精确允许集合；复合结论是否同时权衡两侧；
- 对某爻空、破、动、静、旺、衰、生、克的陈述是否有匹配的爻位事实；
- 应期是否来自已选代码候选，是否擅自增加地支或绝对日期；
- 是否混入其他术数流派或禁止术语。

最终断卦首次失败时会收到具体错误并完整纠正一次；第二次仍不合格则返回 `llm_validation_failed`，不会继续使用未经验证的断语。

### 9. 精准度与质量边界

当前系统不能诚实地宣称“精准度很高”。现有自动测试证明的是排盘、来源、事实一致性和反偏置约束，不等于现实事件预测准确率。`implementation_plan.md` 要求的人工黄金卦例与严格留出盲测尚未完成，因此目前没有可报告的用神选择准确率、吉凶方向准确率或应期准确率。

本轮质量控制解决的是可明确证明的系统偏差：教学示意误作占验卦例、普通负面事实可随意支撑凶断、单个卦例结果锚定总体方向、空破与忌神出现被机械当凶。规则层已显式区分原始、派生、有效事实，并覆盖变爻与伏神旺衰、空破、三合阻滞、卦变、进退、反伏及随鬼入墓等核心条件；遇条件冲突仍由裁决层保留。只有完成独立人工标注、训练/调参集与盲测集隔离、按占类报告方向混淆矩阵后，才能评估经验准确率。

### 10. 为什么完整断卦比排盘慢

确定性部分很快。当前求财示例的一次本机测量：

| 阶段 | 耗时 |
|---|---:|
| 历法、排盘、规则 | 约 10 ms |
| 原文检索 | 约 37 ms |
| 上下文与提示词构建 | 约 6 ms |
| 本地合计 | 约 53 ms |

当前断卦阶段不再把全部路由结果直接塞给模型：强制规则出处之外，只保留少量通用规则、占类规则、事实规则和一个最相关完整卦例；结构化 Schema 由 Provider 协议传递，不在用户消息中重复一份。标准求财样例的断卦消息已从约 3.1 万字符压缩到 1.5 万字符以内。完整流程包含占类与视角判定、最终断卦两次模型调用；最终断卦未通过校验时会增加一次纠正调用，且每次调用分别受 `LLM_TIMEOUT_SECONDS` 限制。`POST /api/v1/chart` 不调用模型，通常可立即返回，但不会给出用神、元忌或应期。

## API

| 方法 | 路径 | 行为 |
|---|---|---|
| `GET` | `/` | 原生 HTML/JavaScript 页面 |
| `GET` | `/healthz` | 健康检查 |
| `POST` | `/api/v1/chart` | 仅执行确定性历法、排盘和通用规则计算，不需要知识库或 LLM 凭据，不判定用神 |
| `POST` | `/api/v1/divinations` | 接收用户所选用神、模型判占类、代码定位与计算、检索原文、模型断卦并校验后返回 |
| `GET` | `/api/v1/sources/{source_id}` | 返回稳定引用对应的本地原文 |

用神两现时由系统按《两现章》标记默认主用爻，并让全部候选进入独立裁决与应期；候选方向不能统一时输出“需再占”，不要求用户选择爻位。若用户所选六亲在当前排盘、月建、日辰、变爻和本宫伏神中均无法定位，接口返回带再占建议的 HTTP 409。知识库或当前 Provider 配置不可用时会明确返回 503，不会切换供应商。

原始语料位于 `zengshan_buyi/`，构建产物写入 `data/generated/`，构建过程不会修改原始 Markdown。当前构建结果为 141/141 章、2,251 个段落、308 个实际六爻图例和 191 个编辑性段落；20 条非致命解析警告记录在 `data/generated/knowledge_build_report.json`。

python scripts/build_knowledge_base.py
python scripts/extract_examples.py

## 明确限制
