## 总评

您的“三阶段”划分是正确的，但第二步不应把规则分批塞进 LLM。**当前系统的大多数基础事实本来就是 Python 计算的，并不依靠检索；真正问题是规则覆盖不全、部分实现背离原文、用神依赖 LLM、以及代码提前用启发式规则限定吉凶。**

因此，当前系统可以评价为：

- **排盘：基础扎实，但尚未达到经过原典逐项认证的程度。**
- **事实层：架构方向正确，但远未完整，且混入了推断和吉凶裁决。**
- **断卦层：约束较强，但输入它的裁决证据本身不完整，因此不能可靠代表《增删卜易》。**

以下结论以当前含未提交改动的工作区为准，包括未跟踪的 `app/rules/outcome.py`。我没有修改任何仓库文件。

## 一、当前系统到底怎样运行

### 1. 排盘接口

`POST /api/v1/chart` 的流程是：

1. 根据输入年月日时生成月建、日辰、旬空、六神等历法信息。
2. 根据六个爻值计算主卦、变卦、八宫、世应、纳甲、五行、六亲、伏神。
3. Python 规则引擎计算旺衰、空破、冲合刑、部分动变及格局事实。
4. 直接返回结果。

这里**没有 LLM、没有知识库检索，也不确定用神**。调用链见 `app/divination/service.py:157-182`、`app/chart/chart_builder.py:32-202`、`app/rules/engine.py:21-65`。

### 2. 完整断卦接口

`POST /api/v1/divinations` 会继续执行：

1. 精确读取九个固定原文段落。
2. 第一次 LLM 只看用户问题和这些段落，输出占类、对象、取世爻还是某个六亲。
3. Python 根据该结果定位具体用神爻，再计算元神、忌神、仇神、裁决证据和应期候选。
4. 检索补充理论及一个卦例。
5. 第二次 LLM 输出结构化断语。
6. 校验失败时允许纠正一次，再失败则报错。

见 `app/divination/service.py:184-234`。

### 3. 当前“文件检索”究竟检索什么

它不是向量检索，分为四种：

| 用途 | 当前方式 |
|---|---|
| 第一次 LLM 选用神 | 精确读取九个固定 `source_id`，不是搜索 |
| Python 已生成事实的出处 | 按事实携带的 `rule_source` 精确读取 |
| 补充理论 | 固定章节、占类、事实标签、FTS5 四阶段 |
| 卦例 | 按占类、事实标签、卦名、用神、问题文字打分，只发送一个 |

服务层实际额度为固定理论 6、占类 8、事实标签 8、FTS 3，总计最多 36 段，见 `app/divination/service.py:326-429`。README 写的是 20/18/18/8，与实现不一致，见 `README.md:149-161`。

此外，FTS 实际使用的是占类名称，不是用户原问题；真正的问题文字只参与卦例评分。

所以准确说法是：

> **事实不是通过检索“发现”的；事实先由代码产生，检索主要为事实补出处，并给最终 LLM 提供理论和卦例。**

## 二、对三个阶段的评价

### 1. 排盘

做得较好的部分：

- 使用精确节气切月，支持 23:00 换日。
- 八宫、纳甲、六亲、世应、动变全部由代码完成。
- 变爻六亲沿用主卦宫五行，这一点正确。
- 伏神从本宫首卦取得。
- 已测试 64 卦宫序与世应、纯乾纳甲、单爻变化和一个伏神案例。

仍不能宣称“已经完全正确”：

- 没有逐卦核对 64 卦每一爻的纳甲、六亲、伏神和原文表格。
- 历法只覆盖少数年份、节气边界和子时样例。
- 静卦仍生成与主卦相同的“变卦”，并可能重复产生主卦、变卦六冲六合事实。
- 一些事实出处只指向章节开头或错误解析的卦例段落，并不能真正支持该事实。
- 全套测试目前并非完全绿色：排除 `tests/knowledge/test_scripts.py` 的导入冲突后有 261 项通过；92% 覆盖率只能证明代码被执行，不能证明符合原典。

**结论：排盘是当前最可靠的部分，但需要原典金标准测试后才能认证。**

### 2. 事实提取

当前覆盖情况如下：

| 原典规则 | 当前状态 |
|---|---|
| 五行生克 | 部分；有基本映射和动变关系，没有完整的所有参与者生克图 |
| 用神、元神、忌神、仇神 | 部分；能标记，但用神入口依赖 LLM，仇神作用基本未进入裁决 |
| 地支冲合刑害 | 冲合刑部分实现；六害未作吉凶规则符合原文，但也没有输出零权重结构事实 |
| 月建、日辰、四时旺衰 | 基本实现，但主要针对主卦明爻 |
| 旬空、月破 | 能识别，未完整实现“空而不空”“破而能用”等效力条件 |
| 暗动、动散 | 有标签；动散只标潜在冲散，尚未形成完整效力规则 |
| 动变、回头生克冲合 | 结构关系已实现，缺化空、化破、化墓、化绝、化鬼 |
| 生旺墓绝 | 只计算本爻相对于日支，未覆盖日墓、动墓、化墓及旺衰前提 |
| 进神、退神 | 能识别支序，但裁决中被过度简化为进即利、退即不利 |
| 飞神、伏神 | 排盘定位有，六种有用、五种无用条件未实现 |
| 三合局 | 不完整；未正确覆盖月日补局、虚一待用、空破待实、入墓待冲 |
| 六冲、六合 | 有结构标签，缺“有气无气、言克不言合”等效力判断 |
| 反吟、伏吟 | 部分实现，伏吟出处和完整条件存在问题 |
| 独发 | 只识别数量，原文明确反对据此直接断祸福 |
| 两现 | 当前采用人为分数排序，与原文大量取空、取破验例冲突 |
| 随鬼入墓 | 明确未实现 |
| 卦变生克墓绝 | 未实现 |

几个关键偏差尤其严重：

- 两现时采用“动爻 +30、旺相加分、空破各 -5”，见 `app/rules/useful_god.py:97-156`；但原文明确指出验例常取旬空、月破，见 `zengshan_buyi/039_两现章.md:7-8`。
- 三合只接受卦内三支齐全且至少两支活动，见 `app/rules/patterns.py:65-84`；原文还要求支持月日补局、虚一待用、暗动、空破填实和入墓待冲，见 `zengshan_buyi/021_三合章.md:21`。
- 生旺墓绝不能机械套表。原文明确规定旺相、休囚、生扶、烈火煎金等会改变“生、墓、绝”的效力，见 `zengshan_buyi/030_生旺墓绝章.md:18-20`。
- 伏神六种有用、五种无用尚未实现，见 `zengshan_buyi/035_飞伏神章.md:37-53`。
- 伏神被选为用神时，`selected_line` 实际仍是飞神位置，而应期模块按该明爻的地支计算，见 `app/rules/timing.py:24-37`，这是实质性错误。

### 3. 断卦

当前第二次 LLM 并不是“看到全部事实和原文后自由判断”。它被明确要求：

> 总体吉凶只能来自 `app/rules/outcome.py` 预先生成的“裁决证据”。

见 `app/llm/prompts.py:61-80`。

而该裁决层已经做了大量启发式判断，例如：

- 用神暗动一律作为有利主证；
- 用神化进一律有利；
- 用神化退一律不利；
- 用有限条件近似“无根”“元神有效”“贪生忘克”。

见 `app/rules/outcome.py:143-205`、`:459-500`。

因此当前真实结构是：

> **不完整的 Python 启发式先限定吉凶，LLM 再负责解释和组织语言。**

此外：

- 用神第一次 LLM 的引用校验只验证“引文确实存在”，不验证该引文是否支持所选六亲，见 `app/divination/validator.py:174-203`。
- “两现并列”不会阻止最终断卦，只会令裁决层暂不裁决。
- 输出可以选择“平”，但“平”没有像“吉、凶”一样要求方向证据，形成校验漏洞。
- `LineProperty` 仅覆盖空、破、动、静、旺、衰、生、克，刑、合、墓绝、进退等自然语言陈述无法严格校验。
- 卦例解析采用“卦图前一段当问题、后一段当断语”，见 `app/knowledge/parser.py:260-392`，容易把理论当问题或截断完整断语。
- 最终只提供一个卦例，容易产生单例锚定。

## 三、建议的新架构

### 1. 保留三阶段，但重新定义第二阶段

| 阶段 | 应负责什么 |
|---|---|
| 排盘 | 完全确定性的历法、卦盘和基础映射 |
| 事实与规则求值 | 完全确定性的事实图、条件规则、例外、优先级和冲突 |
| 断卦 | LLM 只做问题语义理解、受约束的权衡说明和自然语言表达 |

“把规则分批写入提示词”可以用于**离线辅助整理原文**，但不能作为运行时事实引擎。`temperature=0` 也不能保证确定性；当前 OpenAI 兼容层在模型拒绝 temperature 时还会去掉该参数重试，见 `app/llm/_chat_completions.py:39-87`。

### 2. 建立原文规则注册表

每条规则应结构化保存：

- 唯一规则 ID；
- 前提条件；
- 产生的事实或效力；
- 例外与覆盖关系；
- 适用占类；
- 原文出处；
- 对应验例；
- 状态：有效、条件有效、原书废弃。

例如六害应登记为“结构可识别、预测权重为零”，因为原文是“六害全无应验，删之不录”，见 `zengshan_buyi/024_六害章.md:5`。

### 3. 把“事实”分成三层

- **原始事实**：某爻为卯木、与酉相冲、旬空、月破。
- **派生事实**：旺静爻受日冲形成暗动；三合虚一待用。
- **效力结论**：该暗动是元神生用而有效；该六合因无生扶而“言克不言合”。

这样可避免把“出现进神”“出现反吟”直接等同于吉凶。原书大量规则都依赖效力前提，例如六合见 `zengshan_buyi/020_六合章.md:21-55`，随鬼入墓见 `zengshan_buyi/037_随鬼入墓章.md:11`。

### 4. 用神改成“LLM 语义抽取 + 确定性决策表”

LLM 只输出：

- 谁在问；
- 问谁或问什么；
- 人物关系；
- 事项类型；
- 自占还是代占；
- 是否存在语义歧义。

随后由代码依据 `008_用神章` 的完整映射及例外决定世爻或六亲。姐丈、妹夫、表兄弟等例外必须进入规则表，见 `zengshan_buyi/008_用神章.md:5-17`。

无法唯一判断时应要求澄清，不应让模型猜测。两现时保留全部候选和各自条件，不再用固定分数强行选一爻。

### 5. 原文不再“搜索决定规则”

运行时应根据已触发的 `rule_id` 精确取得原文。宽泛检索只用于：

- 找占类专论；
- 找完整卦例；
- 给用户展示延伸阅读。

这样事实、效力和出处永远一一对应，不会出现“小标题支持复杂算法”的情况。

### 6. 卦例改为结构化先例

卦例应人工或半自动整理为完整记录：

- 原问、日期、主变卦；
- 用神；
- 所有关键规则；
- 规则优先顺序；
- 原断；
- 实际应验；
- 哪些规则被原作者否定。

检索应按“当前激活的规则链”匹配，而不是主要按卦名和关键词。应提供多个不同结果的相关实例，防止单例投票；卦例只能帮助解释规则如何应用，不能替代规则。

### 7. 验收标准

优先完成：

1. 64 卦逐爻金标准，以及 64 卦 × 64 种动爻组合的 4096 个变卦不变量测试。
2. 60 甲子、旬空、十二月建和全部交节边界测试。
3. 按 `008–039` 各规则章建立逐条规则测试。
4. 修复伏神实体模型和应期计算。
5. 删除两现人为评分及当前不完整吉凶启发式。
6. 将书中卦例转为“规则链回归测试”，而不是只测试最终文案。

最终应做到：**同一排盘和同一语义化问题必然产生完全相同的事实图、规则命中、冲突和候选结论；LLM 即使换模型，也只能改变措辞，不能改变事实与允许的吉凶范围。**
