# 基于《增删卜易》的六爻占卜系统实施计划

## 1. 项目目标

构建一个带简洁网页前端的六爻占卜系统。用户输入所占之事、公历年月日时，并依次选择六个爻的状态；系统使用确定性代码完成历法换算、排盘、用神和卦象事实计算，再让大语言模型严格依据 `zengshan_buyi/` 中的《增删卜易》原文完成断卦。

系统必须遵循以下原则：

1. 排盘、干支历、五行关系和规则事实全部由代码计算，不允许交给 LLM。
2. LLM 只能依据系统提供的排盘事实和检索到的《增删卜易》原文推断。
3. 每个主要判断都必须给出原文章节和段落引用。
4. 如果原文证据不足或规则存在歧义，系统应明确说明，不能使用模型自身的六爻知识补足。
5. 前端只满足输入和展示需要，不引入 UI 框架、动画、主题或装饰。

## 2. 首版功能范围

### 2.1 用户输入

网页包含以下控件：

- 所占之事：必填文本框。
- 占问类型：可选下拉框；用户不选择时，由系统根据问题给出候选分类，存在歧义时要求用户明确选择。
- 占问关系：自占或代占；代占时填写被占者与用户的关系，用于确定用神。
- 公历年、月、日、时：分别输入，均为必填。
- 时区：首版固定为 `Asia/Shanghai`，页面明确显示。
- 六个爻：从初爻到上爻依次排列，每爻四选一：
  - 老阴 `6`：阴爻，动爻，变阳；
  - 少阳 `7`：阳爻，静爻；
  - 少阴 `8`：阴爻，静爻；
  - 老阳 `9`：阳爻，动爻，变阴。
- “排盘并断卦”提交按钮。

前端必须明确标注“初爻在下、上爻在上”。传给后端的数组始终按初爻至上爻排列，避免视觉顺序造成误解。

### 2.2 输出内容

结果页面按以下顺序显示：

1. 输入摘要：所占之事、公历时间、时区、占问关系。
2. 历法信息：月建、日柱、旬空、六神起法所需日干。
3. 主卦和变卦：卦名、所属卦宫、宫五行、六合/六冲、游魂/归魂等。
4. 六爻排盘表：六神、伏神、六亲、纳甲干支、阴阳、世应、动静、变爻。
5. 用神：采用的用神、元神、忌神、仇神及选择依据。
6. 卦象事实：旺衰、空破、生克冲合、墓绝、进退、回头生克等代码计算结果。
7. 断卦结论：吉凶趋势、关键依据、条件和限制。
8. 应期候选：候选时间及规则依据；证据不足时不输出确定日期。
9. 原文依据：章节名、稳定段落编号和引用原文。

首版使用单体应用，避免不必要的部署复杂度：

## 5. 数据契约

### 5.1 占卦请求

```json
{
  "question": "本次求财是否可成？",
  "category": "求财",
  "subject": {
    "mode": "self",
    "relation": null
  },
  "calendar": {
    "year": 2026,
    "month": 7,
    "day": 17,
    "hour": 9,
    "timezone": "Asia/Shanghai"
  },
  "lines": [7, 8, 6, 9, 7, 8]
}
```

验证规则：

- 问题去除首尾空白后不能为空，并设置合理长度上限。
- 年月日时必须组成真实存在的公历时间。
- `hour` 范围为 0 至 23。
- `lines` 必须恰好六项，每项只能是 `6`、`7`、`8`、`9`。
- 数组索引 0 永远代表初爻，索引 5 永远代表上爻。
- 代占必须提供关系；自占不得伪造关系字段。
- 后端不信任前端计算结果，所有卦象均重新计算。

### 5.2 内部排盘结果

排盘对象至少包含：

- 原始公历时间和换算依据；
- 月建、日柱、旬空；
- 主卦、变卦、上下卦、卦宫、宫序、宫五行；
- 六冲、六合、游魂、归魂；
- 六条爻的阴阳、动静、纳甲、地支五行、六亲、六神、世应；
- 动爻对应的变爻；
- 伏神；
- 可追踪的事实标签列表。

每条事实使用稳定 ID，并包含事实类型、相关爻位、计算参数、结果和规则来源。例如：

```json
{
  "id": "fact-0012",
  "type": "MONTH_BREAK",
  "line": 3,
  "value": true,
  "evidence": {
    "month_branch": "子",
    "line_branch": "午"
  },
  "rule_source": "034_月破章:p0003"
}
```

## 6. 确定性历法模块

用户只输入公历年月日时，系统负责：

1. 校验公历日期和小时。
2. 按 `Asia/Shanghai` 解释输入时间。
3. 依据节气交界计算月建，不按农历初一切月。
4. 计算日柱、日干和旬空。
5. 根据日干排六神。
6. 保存使用的节气时刻和换算结果，便于审计。

实现时优先选择成熟且许可证合适的历法库；以项目测试夹具锁定结果，避免库升级导致静默变化。首版固定以 23:00 为子初换日边界，并将该行为做成配置项。处于节气或换日边界附近时，结果页显示换算说明。

## 7. 排盘引擎

### 7.1 基础常量

从《增删卜易》基础章节整理并人工复核：

- 八卦二进制表示；
- 六十四卦名称；
- 八宫归属和宫序；
- 世应位置；
- 内外卦纳甲地支；
- 宫五行与六亲关系；
- 六神次序；
- 六冲、六合、游魂、归魂；
- 本宫首卦及伏神规则。

这些规则以代码或版本化结构数据保存，不在运行时让 LLM 从文本临时提取。

### 7.2 排盘步骤

1. 将 `6/8` 转为阴爻，将 `7/9` 转为阳爻。
2. 根据六爻阴阳识别主卦。
3. 反转 `6/9` 的阴阳，识别变卦。
4. 确定主卦所属八宫、宫序、宫五行和世应位置。
5. 为主卦逐爻装纳甲地支和五行。
6. 根据宫五行计算六亲。
7. 根据日干自初爻向上排六神。
8. 标记动爻，并计算其变爻纳甲、五行和六亲。
9. 用神不上卦时，根据本宫首卦计算伏神。
10. 输出完整且不可变的结构化排盘对象。

### 7.3 排盘验证

- 用 `003_八宫图章.md` 覆盖全部六十四卦的卦宫、世应、纳甲和六亲。
- 从各章卦例抽取带日期和动爻的案例，验证主卦、变卦、六神、旬空及伏神。
- 六十四个静卦和所有可能的单爻变化必须有参数化测试。
- 排盘核心不得依赖网络或 LLM。

## 8. 《增删卜易》知识库

### 8.1 语料分层

将 141 个 Markdown 章节处理为：

- 基础规则：八宫、纳甲、六亲、世应、动变；
- 通用断卦规则：用神、元忌、生克、旺衰、旬空、月破等；
- 各门类规则：天时、求财、婚姻、疾病等；
- 卦例：问题、日期、卦盘、断语和应验结果；
- 编辑性文字：`[乾按]`、`[提要]`等，保留但默认不作为核心规则依据。

`野鹤曰`和`觉子曰`纳入主要依据。不同段落存在修正或例外时，不强行合并为一条绝对规则，而是保留适用条件和原始出处。

### 8.2 稳定引用

每个可引用段落生成稳定 ID：

```text
008_用神章:p0001
032_各门类应期总注章:p0007
076_求财章:example0003:judgement
```

知识库记录：

- 章节编号和标题；
- 段落 ID；
- 原文；
- 内容类型；
- 主题标签；
- 规则标签；
- 适用占类；
- 源文件路径和内容哈希。

引用必须返回本地原文，不能让 LLM 自行复述后冒充引文。

### 8.3 检索

检索采用以下优先级：

1. 固定取用：凡例、用神及通用方法论。
2. 占类路由：按求财、疾病、婚姻等选择对应章节。
3. 事实标签路由：旬空、月破、回头克等事实触发对应章节。
4. 卦例特征匹配：按占类、用神、卦名、动爻和事实标签评分。
5. SQLite FTS5：仅作为问题关键词和原文检索的补充。

首版不使用向量数据库，避免语义相似但规则不适用的内容进入上下文。

## 9. 用神与规则事实引擎

### 9.1 用神

将 `008_用神章.md`、`039_两现章.md`、`035_飞伏神章.md` 及各门类章节整理为带出处的规则表。处理顺序：

1. 根据明确占类和关系直接选择用神。
2. 处理自占与代占差异。
3. 处理用神两现。
4. 处理用神不上卦及伏神。
5. 仅在问题分类确实有语义歧义时调用 LLM 提供候选；最终必须由用户选择或由确定性映射确认。

### 9.2 事实标签

分批实现并测试：

- 五行生克；
- 地支冲、合、刑、害；
- 月建、日辰作用；
- 四时旺相和休囚；
- 旬空、月破；
- 暗动、动散；
- 动变生克冲合；
- 回头生、回头克；
- 生旺墓绝；
- 进神、退神；
- 飞神、伏神；
- 三合局；
- 六冲、六合；
- 反吟、伏吟；
- 独发、两现；
- 随鬼入墓等书中明确规则。

规则引擎只产生事实和适用条件，不直接输出自由文本吉凶结论。

## 10. 应期模块

根据 `032_各门类应期总注章.md` 及相关章节生成候选，不直接承诺唯一日期。候选规则包括：

- 静爻逢值、逢冲；
- 动爻逢值、逢合；
- 旬空填实、冲空；
- 月破填实、逢合；
- 入墓冲开；
- 六合待冲；
- 太旺待墓或冲；
- 衰绝待生旺；
- 进神、退神的值、合、冲；
- 克神受制或受生；
- 远事按年月、近事按日时。

每个候选必须包含触发条件、候选地支或时间范围、置信限制和原文引用。LLM 只能在候选中结合问题远近做选择，不能自行计算新候选。

## 11. LLM Provider 与约束

### 11.1 Provider 接口

定义统一接口，使业务层与供应商无关：

```python
class LLMProvider(Protocol):
    async def generate_structured(
        self,
        messages: list[Message],
        response_schema: type[BaseModel],
    ) -> BaseModel: ...
```

实现：

- OpenAI；
- Azure OpenAI；
- Anthropic。

应用启动时只初始化 `LLM_PROVIDER` 指定的一个实现。缺少当前 Provider 所需配置时立即报错，不静默切换。

### 11.2 模型输入

模型只接收：

- 用户问题和已确认占类；
- 结构化排盘；
- 已选择用神；
- 事实标签；
- 应期候选；
- 本次检索到的原文段落；
- 严格的结构化输出 Schema。

系统提示明确禁止：

- 重新排盘或修改代码事实；
- 使用未检索到的六爻知识；
- 引入其他流派、神煞或现代口诀；
- 生成不存在的章节、引文或应期；
- 将编辑按语伪装为野鹤原文。

### 11.3 结构化输出

LLM 返回：

- 总体结论；
- 用神分析；
- 月建日辰分析；
- 动爻和元忌分析；
- 特殊格局；
- 应期选择；
- 风险和不确定性；
- 每条判断引用的 `fact_id` 和 `source_id`。

服务端根据结构化结果渲染最终中文，不接受无法解析的自由格式回答。

## 12. 事实与引用校验

返回用户前执行：

1. 检查所有 `fact_id` 是否存在于本次排盘。
2. 检查引用的 `source_id` 是否存在于本次检索结果。
3. 检查引文是否与知识库原文完全一致。
4. 检查应期是否属于代码生成的候选。
5. 检查结论是否错误声称某爻空、破、动、旺、衰或生克。
6. 检查是否出现未授权术语和其他流派规则。

校验失败时，将具体错误和原始上下文交给模型纠正一次。第二次仍失败则返回明确错误，不输出未经验证的断卦。

## 13. 简洁前端

### 13.1 页面结构

仅实现一个页面：

```text
标题
所占之事
占问类型
自占/代占及关系
公历年 月 日 时
初爻 [四选一]
二爻 [四选一]

五爻 [四选一]
上爻 [四选一]
[排盘并断卦]

```

样式只包含可读性所需的间距、表格边框、字体和错误颜色。不使用图标、图片、卡片阴影、渐变、动画或响应式设计框架。

### 13.2 前端行为

- 浏览器只负责收集输入和调用 API。
- 提交前检查必填项及六爻是否全部选择。
- 请求期间禁用按钮，避免重复提交。
- 后端错误原样转为可理解的中文错误信息。
- 排盘表由后端结构化数据渲染。
- 引用可展开查看完整原文及源章节。
- 不把 API 密钥发送到浏览器。

## 14. HTTP API

首版接口：

| 方法 | 路径 | 用途 |
|---|---|---|
| `GET` | `/` | 返回简洁前端 |
| `GET` | `/healthz` | 健康检查 |
| `GET` | `/api/v1/categories` | 返回支持的占类和关系字段 |
| `POST` | `/api/v1/chart` | 仅执行历法换算和排盘，不调用 LLM |
| `POST` | `/api/v1/divinations` | 完整排盘、检索、断卦和校验 |
| `GET` | `/api/v1/sources/{source_id}` | 返回本地原文引用 |

`/chart` 用于独立验证排盘，也便于在 LLM 不可用时仍检查卦象。

## 18. 首版明确不做

- 用户账户、登录、权限和云端历史记录；
- 支付；
- 自动摇卦或随机起卦；
- 农历日期输入；
- 地理位置和真太阳时；
- 语音、图片或聊天式多轮界面；
- React/Vue 等前端框架；
- 向量数据库；
- 模型微调；
- 由 LLM 排盘或自由补充六爻规则；
- 与《增删卜易》无关的神煞和其他流派规则。

## 19. 开始实施前采用的默认约定

1. 公历时间按中国标准时间 `Asia/Shanghai` 解释。
2. 子初换日默认使用 23:00，并允许通过环境变量调整。
3. 六爻请求顺序固定为初爻至上爻。
4. 首版必须填写所占之事；占类可由用户选择。
5. 前端不保存 API 密钥，也不提供 Provider 配置界面。
6. 《增删卜易》原始 Markdown 保持不变；所有索引和结构化产物写入生成目录。
7. `[乾按]`和`[提要]`仅作为补充材料，不作为核心判断的唯一依据。
8. 首版不追求一次覆盖所有复杂规则；但未实现的规则必须明确标记，绝不能交由 LLM 自行计算。
